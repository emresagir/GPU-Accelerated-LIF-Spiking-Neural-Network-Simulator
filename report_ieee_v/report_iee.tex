\begin{abstract}
    This paper presents a GPU-Accelerated simulator that is based on a discrete-time LIF (Leaky Integrate and Fire) neuron model and a synaptic model.
    Simulator is written in C and CUDA, and validated against the Torch reference to ensure correctness.
    Four models are created in total. C and Torch versions, which run on the CPU; Basic and Optimized versions for the GPU. 
    The optimized version consists of three different optimization techniques: Kernel fusion, shared memory tiling, and vectorized load.
    From the results, the optimized GPU version achieved 3.3x speed-up compared to the basic GPU version, 112x compared to the CPU version.
    When the neuron count got larger, the optimized and basic GPU versions converged on the speed-up graph; this is because the models hit the memory-bandwidth limit.
\end{abstract}

\section{Background}

\subsection{Neuron Model}
The discrete-time implementation of the Leaky Integrate-and-Fire (LIF) model 
as discussed by Stan and Rhodes \cite{stan2024learning} allows for 
efficient sequence modeling in SNNs.

The parameters in the equations are: \\

\begin{tabular}{r l}
  $u[t]$ & Membrane voltage (potential) at time step $t$. \\[0.5em]
  $\beta$ & Membrane decay factor. \\[0.5em]
  $s[t]$ & Spike indicator. \\[0.5em]
  $\theta$ & Firing threshold voltage. \\[0.5em]
  $i[t]$ & Input current.
\end{tabular}
\\

The membrane potential update is being done with the equation~\ref{eq:lif_integration}. 

The Reset mechanism is given in equation~\ref{eq:lif_reset} as soft reset. 
Instead of resetting to a default value, 
the membrane voltage is being calculated directly.
This prevent warp divergence if the hard reset chosen, 
because the if condition structure will be introduced.

Lastly, spike generation is being calculated with equation~\ref{eq:lif_spike}. 
The spike is being decided with comparison between membrane voltage and threshold.


% Membrane Potential Update (Integration)
\begin{equation}
u[t] = \beta u[t-1] + (1 - \beta) i[t]
\label{eq:lif_integration}
\end{equation}

% Reset Mechanism (Soft Reset)
\begin{equation}
u[t] \leftarrow u[t] - s[t-1]\theta
\label{eq:lif_reset}
\end{equation}

% Spike Generation (Heaviside Step Function)
\begin{equation}
s[t] = \Theta(u[t] - \theta) = 
\begin{cases} 
1, & \text{if } u[t] > \theta \\
0, & \text{otherwise}
\end{cases}
\label{eq:lif_spike}
\end{equation}

% Definition of Beta (Decay Factor)
where the decay factor $\beta$ is defined by the membrane time constant $\tau$ and simulation time step $\Delta t$:
\begin{equation}
\beta = \exp\left(-\frac{\Delta t}{\tau}\right)
\end{equation}

\subsection{Synapse Model}


Firstly, the SNN simulator could have been without the synapse model. The synapses could have been just the weights without the leakage. 
But for the sake of showing the power of the GPU and increasing complexity, the synapse model was implemented.

The synapse model is decided as current-based exponential synapse model.
The equation of the exponential synapse model is given in the equation~\ref{eq:discrete_synapse}.\\

Parameters are given for synapse model: \\[0.5em]
\begin{tabular}{r l}
  $\alpha$ & Synaptic decay factor ($e^{-\Delta t / \tau_s}$). \\[0.5em]
  $w_{ji}$ & Weight from neuron $j$ to $i$. \\[0.5em]
  $s_j[t]$ & Spike indicator from presynaptic neuron $j$. \\[0.5em]
  $g_i[t]$ & Postsynaptic current state (decaying memory). \\[0.5em]
  $u_i[t]$ & Membrane potential of neuron $i$. \\[0.5em]
  $\beta$ & Membrane decay factor ($e^{-\Delta t / \tau_m}$). \\[0.5em]
  $\theta$ & Spike threshold.
\end{tabular}


\begin{equation}
g_i[t] = \alpha\, g_i[t-1] + \sum_j w_{ji}\, s_j[t]
\label{eq:discrete_synapse}
\end{equation}

Connecting this synapse model with the neuron model is 
resulted this equation~\ref{eq:updated_neuron_model}.
The input current is changed with the Postsynaptic current state, 
which is calculated in equation~\ref{eq:discrete_synapse}.

\begin{equation}
u_i[t] = \beta u_i[t-1] + (1 - \beta) g_i[t] 
\label{eq:updated_neuron_model}
\end{equation}

Current implementation of the model is dense. 
The sparsity can be added later, but will introduce a high level of complexity.

\section{Proposed Method}
From the previous section, we know the spiking neural network calculation depends on repeatedly performing the given calculations over a specified number of timesteps. 
Even though different timesteps have data dependencies, different neurons can be calculated on different threads, which gives us the opportunity to exploit the GPU's parallel computing power.

My approach basically involves having different models do the same work, and comparing their performance with different parameters (neuron count, timestep).

Firstly, creating a base with the CPU Model, then having a reference for its correctness by using the Python torch library.
Secondly, creating a basic GPU model, which will give us a baseline for the GPU. 
Lastly, optimizing the GPU Model. This process will provide four different models that can be compared. 

Every model will be compared with a small test set to be sure that all the calculations are correct.

At the end, the performance comparison will be done between the models to understand the speed-up. And will compare the work with other related works.


\section{CPU Model}

\subsection{Implementation}

The CPU implementation is done with the given neuron and synapse model definitions
in the previous chapters. It is written in C. The main time loop is given in Figure~\ref{fig:C_impl}. 
  
  For the synaptic update, the external input current is calculated with the given external spikes and weights, then, with a second for loop, 
  the recurrent connections are accumulated into the input current; lastly, the synaptic decay is being applied.
  
  For the neuron update, a soft reset is applied if there was a spike in the previous timestep;
  then the leakage and integration of the input current are applied. Lastly, a spike 
  is recorded if the membrane potential is higher than the threshold. \\


  \begin{figure}[h]
\centering
\begin{lstlisting}
/* Time loop */
    for (int t = 0; t < T; ++t) {

        /* Synapse update */
        for (int i = 0; i < N; ++i) {
            float input_current = 0.0f;
            input_current += ext_spikes[t*N + i] * ext_weight;
            for (int j = 0; j < N; ++j) {
                input_current += Wflat[i*N + j] * s_prev[j];
            }
            g[i] = alpha * g[i] + input_current;
        }
        /* Neuron Update */
        for (int i = 0; i < N; ++i) {
            u[i] -= s_prev[i] * theta;
            u[i] = beta * u[i] + (1.0f - beta) * g[i];
            s[i] = (u[i] > theta);
        }
        /* Copy spikes */
        for (int i = 0; i < N; ++i) {
            s_prev[i] = s[i];
        }
\end{lstlisting}
\caption{Time loop of the CPU implementation}
\label{fig:C_impl}
\end{figure}

\section{Basic GPU Model}

\subsection{Implementation} \label{basic_problems_mentioned}
The Basic GPU Model implementation is done after the CPU model, to have a middle ground before optimizing this version.

For this basic version, the kernels are separated. One is for the synapse calculation, the other is for the neuron.

The synapse kernel is given at the Figure~\ref{fig:Synapse_Basic_GPU_impl}.
With the GPU version it is clearly visible that now there is only one for loop, compared to two for loops of the CPU model. 
Still the biggest bottleneck is this the recurrent calculation loop.

  \begin{figure}[h]
\centering
\begin{lstlisting}
  __global__ void synapse_kernel(...)
  {
      int i = blockIdx.x * blockDim.x + threadIdx.x;
      if (i >= n) return;

      // Accumulation
      float acc = (float)ext_step[i] * ext_w;

      // Adding recurrent input from all previous spikes.
      int row = i * n;
      for (int j = 0; j < n; ++j)
          acc += W[row + j] * (float)s_prev[j];

      // Decay 
      g[i] = alpha * g[i] + acc;
  }
\end{lstlisting}
\caption{Synapse Kernel of the Basic GPU implementation}
\label{fig:Synapse_Basic_GPU_impl}
\end{figure}

The neuron kernel is given at the Figure~\ref{fig:Neuron_Basic_GPU_impl}. After the kernel indexing is done, the membrane potential being updated and the spikes are being recorded.


  \begin{figure}[h]
\centering
\begin{lstlisting}
  __global__ void neuron_kernel(...)
  {
      int i = blockIdx.x * blockDim.x + threadIdx.x;
      if (i >= n) return;

      // Membrane Update (Soft Reset)
      float ui = u[i] - (float)s_prev[i] * thresh;
      ui = beta * ui + (1.0f - beta) * g[i];

      // Spike generation
      uint8_t si = (ui > thresh) ? 1 : 0;
      u[i] = ui;
      s[i] = si;
  }
\end{lstlisting}
\caption{Neuron Kernel of the Basic GPU implementation}
\label{fig:Neuron_Basic_GPU_impl}
\end{figure}

In this naive version, there are a couple of problems that are apparent. There is two seperate kernel introduces an overhead.
And the inner loop of the synapse kernel reads the full s\_prev for every neuron.
But the biggest problem is that the Weights are being loaded without a coalesced access. 
The weight access is done with W[i*n + j], can be seen in the Figure~\ref{fig:Synapse_Basic_GPU_impl}. 
Consecutive threads are not accessing the consecutive memory addresses. 
This introduces the biggest bottleneck.

\section{Optimized GPU Model}
\subsection{Implementation}
The performance problems of the basic GPU version are previously mentioned in the section~\ref{basic_problems_mentioned}. 

The two kernel design choices were for reducing complexity. These two kernels are fused together to optimize the code.
By fusing these two kernels into one, the kernel launch overhead at every timestep was reduced to half.
Moreover, the $g_i[t]$ Postsynaptic current was written to global memory and then read by the next kernel at every timestep. 
With the fusion, the $g_i[t]$ lives in the registers, and reduces the latency.

Second problem, reading the full s\_prev[] array (previous timestep spike array) for every neuron. 
All neurons use the same data, so with shared memory tiling, the data is being loaded into the shared memory to be used by other threads in the block, making this process faster.

Lastly the biggest bottleneck, the un-coalesced access of the weight matrix. 
The optimal way would be preprocessing the weight matrix with a transpose, and having the access with W[j*n + i]. 
But in this project the focus is using GPU optimization techniques, thus float4 vectorized load is introduced.

In the Figure~\ref{fig:Fused_Optimized_GPU_impl}, the fused kernel is shown with the mentioned optimizations: fused kernel, memory tiling and vectorized load.
The kernel first computes the external contribution of the external spikes and the weights. 
Then, with the memory tiling and the vectorized load the recurrent connections are accumulated into the acc variable.
The tile size was decided as 256, and the block size was also decided as 256; this way, all the threads are loading the data from the global memory to the shared memory tile.
In the figure~\ref{fig:thread and block calc}, this thread and block count calculation is shown. 
Lastly, the decay and the membrane potential update are done.

  \begin{figure}[h]
\centering
\begin{lstlisting}
__global__ void fused_snn_kernel(...)
{
    int i = blockIdx.x * blockDim.x + threadIdx.x;

    __shared__ uint8_t s_tile[TILE_W];

    float ext_contrib = (i < n) ? (float)ext_step[i] * ext_w : 0.0f;

    float acc = ext_contrib;

    const float* W_row = (i < n) ? W + (size_t)i * n : NULL;

    for (int tile_start = 0; tile_start < n; tile_start += TILE_W) {
        int j_load = tile_start + threadIdx.x;
        s_tile[threadIdx.x] = (j_load < n) ? s_prev[j_load] : 0;
        __syncthreads();

        if (i < n) {
            int tile_end = min(TILE_W, n - tile_start);
            int k = 0;
            int tile_end4 = (tile_end / 4) * 4;
            for (; k < tile_end4; k += 4) {
                float4 w4 = *reinterpret_cast<const float4*>(W_row + tile_start + k);
                acc += w4.x * (float)s_tile[k    ];
                acc += w4.y * (float)s_tile[k + 1];
                acc += w4.z * (float)s_tile[k + 2];
                acc += w4.w * (float)s_tile[k + 3];
            }
        }
        __syncthreads();
    }

    if (i >= n) return;

    float gi = alpha * g[i] + acc;
    g[i] = gi; 

    float ui = u[i] - (float)s_prev[i] * thresh;
    ui = beta * ui + (1.0f - beta) * gi;

    uint8_t si = (ui > thresh) ? 1 : 0;
    u[i] = ui;
    s[i] = si;
}
\end{lstlisting}
\caption{Fused Kernel of the Optimized GPU implementation}
\label{fig:Fused_Optimized_GPU_impl}
\end{figure}

\centering
\begin{lstlisting}
    int threads = TILE_W;   /* == 256 */
    int blocks  = (N + threads - 1) / threads;
\end{lstlisting}
\caption{Thread and Block calculations}
\label{fig:thread and block calc}
\end{figure}

\section{Results and Performance Evaluation}
First of all, the sbatch script was edited to be an automated one to see the performance with different N (neuron) and T(timestep) values.
And the N and the T variable names are changed into GRID\_N and STEPS\_T. 
This is done in order to fix the problem with the usage of the nvcc -D flag.
The sbatch is compiling the source file as given in the Figure~\ref{fig:NVCC_compile}.
With the automated script, different N values are given to the compiler to overwrite the defined values.
At the end, all the speed results are saved into a file. 

  \begin{figure}[h]

With a Python script, the speed result file is read, and the speed values with different T and N values are plotted as can be seen in the Figure~\ref{fig:execution_w_diff_T_and_N}.
Except for the two divergent points in the graph, the results are expected. The speed-up is eminent in the beginning and slowly decreases while the scaling increases. 
And the points nearly merge at the highest Neuron Value. 

The merge at the end is the result of the memory bandwidth limit. 
As I mentioned before, the biggest bottleneck is the uncoalesced access to the weight matrix.
This is being optimized with the float4 vectorized load. 
But this only made the transitions more efficient, loading 16 bytes in just one transaction. 
Basically, using the memory bandwidth more efficiently, 
But when the neuron count increased, the block count also increased. When N is 256, there is only 1 block; when N is 16384, there are 64 blocks which does parallel transaction requests to/from memory.
Which saturates the memory bandwidth without the need for vectorized efficient loads.

\section{Discussion}
The results show a big speed-up with the optimized GPU Model over the CPU model.
The speed-up can reach 112.7× when the parameters saturate the GPU.
But this big speed-up is not visible across the entire parameter space. 
Especially when the neuron count is minimal (N=256), the GPU setup and kernel launch overheads, 
makes the GPU slower than the CPU counterpart.

\subsection{Analysis of Optimization Techniques}

Three optimization techniques is introduced for a higher efficieny and achieve more optimized code. 

\begin{itemize}
    \item \textbf{Kernel Fusion:} To prevent the kernel launch overhead, kernel fusion is being used. The two kernels of the Neuron and Synapse calculations merged into one. This also resulted in a lower global memory R/W as the $g_i[t]$ started to live in the registers. 
    \item \textbf{Shared Memory Tiling:} The basic GPU version was reading the whole $s_{prev}$ array for every neuron. By loading this array in the shared memory tile, the data can be read by all threads in the same block.
    \item \textbf{Vectorized Loads:} The vectorized load of float4 helped the optimized version to use the memory bandwidth more efficiently resulting a faster execution in the beginning and the mid range of parameters.
\end{itemize}

\subsection{Memory Bandwidth Bottleneck and Saturation Point}

The Optimized and the Basic versions converge at the high number of neurons (N = 16386), it is visible from the Figure~\ref{fig:execution_t_all}.
This is expected as when N = 16386, the recurrent array of the weights is 1.07 GB. 

$$\text{Memory} = 16384^2 \times 4 \text{ bytes} \approx 1.07 \text{ GB}$$

At this point, the simulation is no longer compute-bound; it is bandwidth-bound. From the Figure~\ref{fig:speed_factor}, it is understandable that optimization techniques lost their edge when the execution becomes memory-bandwidth bound.

\subsection{Matrix Transposition and Sparsity}

The introduction of the vectorized loads did help to achieve more efficient memory transfers for low to mid-range. 
But the real problem is that the uncoalesced memory alignment of the weight array. This is not solved by the vectorized loads. To achieve a more optimized version, the matrix needs to be transposed. Thus enabling consecutive threads to access consecutive weights. 
At the end, this solution was not implemented to stay on topic.

Another optimization point would be introducing sparsity. Could enable more optimized memory access, changing the $O(N^2)$ to $O(E)$, where E is the number of non-zero synaptic connections. But I did not implemented because of the complexity it would bring, and it is off-topic.

\subsection{Related Works}

CARLsim6 \cite{niedermeier2022carlsim}, and GeNN \cite{yavuz2016genn} are some of the most popular GPU-accelerated simulators.
The current state of this work is well below what those simulators are achieving, but still a solid proof-of-concept.

The CARLsim6 is a biologically detailed and large-scale SNN simulator, which is being used in biologically plausible simulations.
It provides a C/C++ API, with it user can manage neuron groups and the connectivity patterns. 
This makes it highly generic and can be used on the majority of the GPUs.

The GeNN is taking another approach; instead of giving a run-time framework, it provides a code generation tool.
The user supplies the parameters and the model definitions, and the generator outputs an optimized code for the specific GPU.
Their results show a speed-up of up to 200x. 

When analyzing the related works, 
It is apparent that they are highly generic, even though their approach is different.

My work shows a solid proof-of-concept. 
For future work, the simulator can be more generic.
By using the proven optimization techniques given in the report, a Python code generator script can be implemented. 
The code generator can take the parameters (weights, connectivity, etc.) and generate an optimized code. 
The GeNN is not specific to Spiking Neural Networks, so optimizations can exploit this specificity. 


The NIR (Neuromorphic Intermediate Representation) \cite{pedersen2024neuromorphic} 
It is an open-source standard and graph-based format that can provide a graph with the needed neuron parameters. 
Thus, while feeding the parameters into the code-generator script, the NIR can be used.
